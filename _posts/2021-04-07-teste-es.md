# QA

# Projeto de Qualidade

## **A qualidade é difícil de definir em termos abstratos.**

- pode ser reconhecido se presente
- algumas propriedades intangíveis que encantam os usuários
- ideal que pode nunca ser completamente alcançado

## **Visualização do usuário**

- é adequado para seu propósito, satisfatório
- depende de como o produto atende às necessidades de seus usuários para as tarefas
- pode ser medido com base na confiabilidade, usabilidade, acessibilidade, disponibilidade,

segurança, exatidão, precisão, etc.

- usuários diferentes terão necessidades diferentes

## Visão de fabricação

- conformidade do produto com os padrões do processo
- depende de quão bem o produto foi construído
- potencial para evitar custos associados ao retrabalho durante o desenvolvimento e após a entrega
- medir usando contagens de defeitos, custos de retrabalho

## Visualização do produto

- vinculado às características inerentes do produto
- um mede e controla as propriedades internas do produto para melhorar o comportamento do produto externo
- exemplos de medidas internas de propriedade do produto
- tamanho, proporção de código / comentários, código, complexidade

Visão baseada em valor

- depende de quanto os clientes estão dispostos a pagar por custo de compensação e qualidade

## Conformidade com os requisitos

- Falta de defeitos: Nº de defeitos / unidade de tamanho
- alta confiabilidade (número de falhas por n horas de operação)
- medido como Tempo médio de falha probabilidade de operação sem falhas em um tempo especificado
- satisfação do usuário / cliente

## Erros e falhas

## conceitos-chave associados à correção do software

- * Falha **: incapacidade de um sistema ou componente de realizar suas funções exigidas dentro dos requisitos especificados
- * Falha ** (defeito / bug): uma etapa, processo ou definição de dados incorreta em um artefato de software
- * Erro **: uma ação humana que produz um resultado incorreto
- Bug / defeito / falha * consequência de um erro humano / erro resulta em não conformidade com os requisitos que se manifesta como * falha * na execução do software

## Exemplos de erros

1. Definição de requisitos defeituosos

2. Falhas de comunicação cliente-desenvolvedor

3. Desvio deliberado dos requisitos de software

4. Erros de design

5. Erros de codificação

6. Não conformidade com a documentação das instruções de codificação

Exemplos de situações de falha

1. O software não faz algo que a especificação diz que deveria fazer

2. O software faz algo que a especificação diz que não deveria fazer

3. O software faz algo que a especificação não menciona

4. O software não faz algo que a especificação não menciona, mas deveria

5. O software é difícil de entender, difícil de usar, lento, ...

6. O software é visto por suas partes interessadas como simplesmente errado

## Importância da Qualidade do Software

- O software é o principal componente dos sistemas de computador (mais de 80% do custo). Usado para
- comunicação
- monitoramento de saúde
- transporte
- Os detores de software  são extremamente caros em termos de:

Impacto econômico

- $ 22 - $ 59 bilhões de dólares perdidos em testes de software inadequados

Desafio de Qualidade de Software

- alta complexidade
- invisibilidade do produto
- oportunidades limitadas para detectar defeitos ('bugs')
- única oportunidade é o desenvolvimento de produtos
- requisito para trabalho em equipe
- necessidade de cooperação e coordenação com outras equipes de desenvolvimento
- necessidade de interfaces com outros sistemas de software
- necessidade de continuar realizando um projeto enquanto a equipe muda
- necessidade de continuar mantendo o sistema de software por anos

Objetivos do SQA

- Assegurar um nível aceitável de confiança de que o software está em conformidade com os requisitos técnicos funcionais
- Assegurar um nível aceitável de confiança de que o software está em conformidade com a programação gerencial e os requisitos orçamentários
- Iniciação e gestão de atividades para a melhoria e maior eficiência das atividades de desenvolvimento, manutenção e SQA de software.

Princípios Gerais de QA

- Saiba o que você está fazendo

- entender o que está sendo construído e como está sendo construído e o que faz atualmente

- saiba o que você deveria estar fazendo

- tendo requisitos e especificações explícitas

- usar um processo de desenvolvimento de software com * análise de requisitos, testes de aceitação, feedback frequente do usuário *

- saiba como medir a diferença

- ter medidas explícitas comparando o que está sendo feito com o que deveria ser feito

- quatro métodos complementares:

- métodos formais

- teste

- inspeções

- Métricas

Ciclo de vida abrangente

- Abordagem abrangente do ciclo de vida preocupada com cada

aspecto do processo de desenvolvimento de produto de software

- conjunto abrangente de objetivos de qualidade
- atributos de qualidade mensuráveis (métricas de qualidade) para avaliar o progresso em direção aos objetivos
- metas quantitativas de certificação para todos os componentes dos processos de desenvolvimento de software.

Atividades

Objetivo: garantir que poucos defeitos, se houver, permaneçam no sistema de software quando entregue.

- Prevenção de defeitos

- evitar que falhas sejam injetadas no software

- eliminação de fontes de erro

- prevenção de falhas ou bloqueio

- Exemplos

- Treinamento

- remova os equívocos

- remover ambigüidades

- conformidade do processo e aplicação do padrão

- Redução de defeitos

- detecta e remove falhas antes que o sistema seja entregue

- Exemplos

- inspeção

- simulação de protótipo

- análise estática

- teste

- contenção de defeitos

- minimizar os efeitos de defeitos (falhas)

- contém falhas locais que não se propagam (tolerância a falhas)

- limita os danos causados por falhas

Verificação:

- "estamos construindo o produto certo?"
- o software deve estar em conformidade com suas especificações
- O processo de avaliação de um sistema ou componente para determinar se os produtos de uma determinada fase de desenvolvimento satisfazem as condições impostas no início dessa fase. IEEE Std 1012-2004

Validação:

- "estamos construindo o produto certo?"
- o software deve fazer o que o usuário realmente precisa.
- O processo de fornecer evidências de que o software e seus produtos associados satisfazem os requisitos do sistema alocados ao software no final de cada atividade do ciclo de vida, resolvem o problema certo e satisfazem o uso pretendido e as necessidades do usuário. IEEE Std 1012-2004

Engenharia de Qualidade de Software

- abrange SQA com atividades relacionadas (planejamento, monitoramento, melhoria)
- abordagem de engenharia para gerenciar as expectativas de qualidade

- estabelecimento de metas de qualidade

- medição do desempenho em relação a essas metas

- análise e melhoria de processos

Componentes principais

- Planejamento de qualidade

- define objetivos de qualidade específicos (com base nas expectativas do cliente, limitações econômicas / de tempo)

- forma a estratégia geral de qualidade (seleciona atividades de controle de qualidade, seleciona medidas de qualidade)

- Avaliação e melhoria da qualidade

- feedback das medições coletadas durante as atividades de QA

- pode solicitar mudanças / ajustes às metas de qualidade, estratégia de QA, processo de desenvolvimento

*Grau em que o produto de software satisfaz as necessidades declaradas e implícitas quando usado sob condições especificadas.* Conjunto de atividades destinadas a garantir que poucas, ou nenhuma, falhas permaneçam no sistema de software quando este é entregue aos seus clientes ou lançado no mercado. Processo que garante que o software desenvolvido atenda e cumpra as metas de qualidade definidas ou padronizadas.
